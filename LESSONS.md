# Lessons Learned

## What I Learned
<!-- Add entries after each session where you learned something new -->

- **XML tags in prompts are a real thing, not just hype.** All three major providers (Anthropic, OpenAI, Google) recommend XML for complex, multi-section prompts. The key insight: XML helps AI distinguish *what kind of content* a section is (rules vs templates vs examples), which markdown headers alone can't do.
- **Hybrid approach beats all-or-nothing.** Don't replace markdown with XML - use both. Markdown headers stay for human readability, XML wraps content blocks for AI parsing. Pattern: header outside the tag, content inside.
- **AI peer review recommendations often over-engineer.** When /ask-gpt and /ask-gemini reviewed the XML plan, they recommended namespaced tags (`tk_rules`), a formal schema file, testing rubrics, and a three-wave rollout. Most of that was unnecessary for 9 small files. Trust your gut when something feels like too much.

- **Vercel AI SDK v6 renamed everything.** `maxSteps` is now `stopWhen: stepCountIs(N)`. `parameters` on tool definitions is now `inputSchema`. `toDataStreamResponse()` is now `toUIMessageStreamResponse()`. Online tutorials and AI training data reference v3/v4 names. Always check the actual type definitions in `node_modules/ai/dist/index.d.ts` when something doesn't compile.
- **Confirm-before-act works with prompt engineering alone.** No need for middleware or multi-step confirmation flows. Put the rule in the system prompt ("NEVER pass confirmApprove: true unless user explicitly confirmed") and require a `z.literal(true)` field on destructive tool schemas. The LLM consistently asks for confirmation first.
- **LLM tools should call services, not HTTP routes.** Calling your own API endpoints from tool functions creates unnecessary network round-trips and auth headaches. Call the service layer directly - you already have the RequestContext.

- **Validate at trust boundaries, not everywhere.** Tool output crosses a network boundary (SDK serialization), so the client can't trust it blindly. `safeParse()` at the consumption point catches issues that `as SomeType` casts hide. Casts lie, runtime validation doesn't.
- **Three reviewers catch different things (again).** Claude found XSS and unsafe casts. GPT found stale deduplication state and unused refs. Gemini found backtick fence issues with LLM-generated mermaid. The overlap was surprisingly small - no single reviewer caught everything.
- **`dangerouslySetInnerHTML` means what it says.** Even when the input comes from a library (mermaid), the *source* of the mermaid syntax is LLM-generated text. That's user-equivalent input. DOMPurify is the standard fix for SVG sanitization.
- **DOMPurify's SVG profile strips Mermaid text labels.** Mermaid v11 uses `<foreignObject>` (HTML-inside-SVG) for text labels and `<style>` blocks for theme colors. DOMPurify's `USE_PROFILES: { svg: true }` strips both because they're not pure SVG elements. Fix: `ADD_TAGS: ["foreignObject", "style"]`. Shapes and arrows survive (pure SVG) but text vanishes - a confusing symptom if you don't know Mermaid's rendering internals.
- **StrictMode double-renders break stateful IDs.** `Date.now()` for element IDs causes collisions because React StrictMode renders twice in dev. `crypto.randomUUID()` is always unique with no timing dependency.
- **Deterministic ordering needs separate arrays.** Pushing to a shared array inside `Promise.all` callbacks creates a race condition - whichever query resolves first writes first. Use separate arrays with fixed-order flattening after `Promise.all`.

- **AI SDK v6 tool part types aren't what the docs suggest.** Tool results arrive as `tool-{toolName}` (e.g. `tool-showTable`), not `dynamic-tool`. The panel logic only checked `dynamic-tool` while the message bubble already used a broader filter (`startsWith("tool-")`). Same codebase, two different assumptions about the SDK. When one component works and another doesn't, compare their filters - the working one probably got it right.
- **Diff the working thing against the broken thing.** When something works in one place but fails in another, comparing the two implementations is faster than reading docs or guessing. In this case, `message-bubble.tsx` (tool indicators worked) vs `page.tsx` (panel didn't) - a 30-second diff revealed the exact line that needed to change.

- **Vitest `globalSetup` vs `setupFiles` matters for Prisma.** Prisma reads `DATABASE_URL` at module import time (when `src/lib/prisma.ts` loads). `setupFiles` runs after imports, so it's too late to swap the URL. `globalSetup` runs before any test file is loaded, so it can inject the right env var before Prisma ever sees it.
- **Deterministic seed scripts don't clean up random orphans.** A delete-then-create seed using known UUIDs only removes rows it created. Test fixtures with random IDs survive re-seeding. If tests create data in a shared DB and cleanup fails (crash, Ctrl+C), those orphans accumulate. Solution: isolate tests in a separate database entirely.
- **Prisma has an AI safety check for destructive operations.** `prisma migrate reset` detects when it's being run by an AI agent and blocks until explicit user consent is passed via the `PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION` environment variable. The value must be the exact text of the user's consent message. Good guardrail, but expect it to interrupt automated workflows.
- **Squash stale migrations early in solo projects.** Two migrations that drifted (modified after being applied) caused a cascade of workarounds (`db push` + raw SQL in test setup, manual constraint scripts, "Known Limitation" docs). Resetting to a single authoritative migration took 20 minutes and eliminated all of it. When there are no other environments depending on the history, a clean slate is simpler than patching drift.

- **Set up `.gitignore` before creating secret files.** If a secret is committed even once, it lives in git history forever - rotating the key doesn't remove it from old commits. Our `.gitignore` excluded `.env`, `.env*.local`, and `.env.test` from day one, so no secrets ever hit the repo. But secrets shown in any tool output (like a Claude conversation or CI logs) should also be treated as compromised and rotated immediately.
- **Rate limiting matters most on AI endpoints.** On an AI-powered app, the most important endpoint to rate-limit is your LLM chat endpoint. A simple in-memory sliding window (no Redis needed at demo scale) took 60 lines of TypeScript and prevents runaway usage.
- **Security headers take almost no effort.** Four lines in `next.config.ts` add X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and Permissions-Policy. No code changes and no dependencies. Skip CSP initially (Next.js inline scripts break strict CSP without nonce configuration) and skip HSTS (Vercel adds it automatically).

- **Check existing FK relations before building AuditLog queries.** The original plan for "Created By" columns proposed a `batchFetchCreators` helper that queried AuditLog for CREATE actions. Code review caught that every entity already has a `createdBy` FK + `creator` relation to User - a simple Prisma `select` join. The AuditLog approach would have been slower and more fragile.
- **Three-model review consensus catches domain bugs.** All three reviewers (Claude, GPT, Gemini) independently flagged that SKIPPED test cases were being counted as PENDING in the aggregation query. Different signal, different meaning - SKIPPED is terminal, PENDING means not yet run. Without the review, users would see inflated "pending" counts.
- **Fetch N+1, return N for truncation detection.** Query `take: 16` but only return 15 rows. If 16 come back, set `isTruncated: true` and tell the user more results exist. Simpler than running a separate COUNT query, and the extra row is discarded before reaching the client.

- **"Documented behavior with no implementation" is a bug category.** The STATUS-GUIDE said "DRAFT = edit freely" but no edit path existed for several entities. The docs and code told different stories. When reviewing, check documented behaviors against actual implementation - not just code quality.
- **Analyze the full state matrix before scoping.** We started thinking Issue #48 was "add DRAFT editing" but walking through every entity x every status revealed a much broader picture (approved typos, DRAFT cancel, test procedure titles, pending test cases). A table of entity x state x allowed operations catches gaps that narrative descriptions miss.
- **"Block if children exist" is safer than cascading for DRAFT cancel.** Cascading makes sense for APPROVED cancel (committed work being retired), but for DRAFT cancel the user hasn't committed to anything yet. Forcing intentional cleanup prevents accidental data loss without adding ceremony.
- **Review findings converge across reviewers.** The DRAFT/APPROVED description redundancy was flagged independently by both UX and Command reviews. When multiple reviewers find the same thing unprompted, fix it immediately rather than deferring.
- **System prompt and schema guards must agree.** If the system prompt references a confirmation field name (`confirmCorrection`), the tool schema must have that field. A phantom reference confuses the LLM - it might try to pass a field that doesn't exist or skip confirmation entirely. Prompt engineering and `z.literal(true)` guards are two layers of the same pattern; they should reinforce each other, not contradict.
- **Error messages should suggest alternatives, not just reject.** When related operations exist (correctTestResult vs reExecuteTestCase), an error from one should point the user to the other. "Cannot re-execute PASSED test case" is a dead end. "Cannot re-execute PASSED test case - use correctTestResult instead" keeps the user moving. Costs one extra sentence, saves a round-trip of confusion.

- **Validation order matters for error quality.** When a service has multiple guards (entity status, target validation, no-op detection), put target-exists and target-status checks before the no-op check. If you check no-op first, a user who passes the same CANCELED parent gets "already under this parent" instead of the more useful "cannot move to a canceled parent." Small reordering, big UX improvement.
- **Return old + new values from structural mutation tools.** When an LLM tool moves an entity, returning only the updated state forces the LLM to remember prior context to compose a "moved from X to Y" message. Returning `previousParentId` alongside the new parent makes the response self-contained. Same principle applies to any correction/transition tool.
- **Re-parenting is simpler than it looks (when the data model is clean).** Moving an entity to a different parent is just a FK update when there are no lineage-based auth rules, no denormalized parent-derived data, no per-parent ordering, and no cached aggregates. Document these assumptions explicitly so future maintainers know what to re-evaluate if the system grows.

- **Frontend tool registration is easy to miss.** We built 3 new backend tools that worked perfectly (279 tests passing, TypeScript clean), but the panel didn't open. Root cause: `UI_INTENT_TOOLS` in `tool-labels.ts` is a name-based allowlist that gates which tool results get pushed to the panel store. Six parallel reviews (code, UX, full, browser, Gemini, GPT) all missed it. Only live user testing caught it. When adding new tools, check the full pipeline from backend execution to frontend rendering.
- **Shell env vars override .env.local silently.** A `~/.bashrc` export of `ANTHROPIC_MODEL=claude-opus-4-6` overrode the `.env.local` setting of Haiku. Next.js does not override environment variables that already exist in the shell. The app was using a more expensive model with lower rate limits without anyone realizing it. After debugging rate limit errors, `echo $ANTHROPIC_MODEL` in the terminal revealed the override instantly.
- **"No frontend changes needed" can be a trap.** The plan correctly noted the `DiagramPayload` type was unchanged (no new Zod schema, no new React components). But the routing layer had a name-based allowlist that DID need updating. "Same type" does not mean "same registration." Always grep for the old tool name to find everywhere it's referenced before declaring the frontend unchanged.

## Mistakes to Avoid
<!-- Add patterns that caused problems so they don't repeat -->

- **Don't micro-tag individual bullets.** Tag sections, not lines. Wrapping every bullet in XML adds noise without helping parsing.
- **Watch for tool output artifacts in reviews.** The Read tool's output wrapper (`</output>`) can look like actual file content. Always verify with raw bytes before "fixing" something that might not be broken.
- **Skill tool expansions can serve stale command versions.** Root cause: old command files in `~/.claude/commands/` (global/home directory) override the project's `.claude/commands/` files. In our case, 9 files were manually copied to the home directory in January before setup.sh existed, then never updated. When slash commands ran, the AI used the global copies instead of the current project versions - missing Staff Engineer Check, Finding IDs, and other updates. Fix: delete global copies (commands should only live in each project folder) and setup.sh now warns if it detects conflicting global files. See issue #52.
- **Prisma CLI reads `.env`, not `.env.local`.** Next.js reads `.env.local` but Prisma CLI only reads `.env`. If your DATABASE_URL is in `.env.local`, create a `.env` with just that variable (already in `.gitignore`).
- **When rotating credentials, check ALL env files.** Secrets can live in `.env`, `.env.local`, and `.env.test`. Updating one but not the others causes confusing auth failures. After rotation, grep all env files for the old value.
- **`create-next-app` rejects uppercase project names.** Scaffold into a temp dir with a lowercase name, then copy the config files into your actual project directory.
- **Don't put Prisma types in Edge-compatible files.** The demo-users module needed to work in Edge Middleware, so it must be pure TypeScript with no Prisma imports. Keep Edge-compatible modules free of Node-only dependencies.
- **PostgreSQL enum renames need the text-cast strategy.** You can't just `ALTER TYPE ADD VALUE` + rename in a single migration - PostgreSQL requires new enum values to be committed before use. Working approach: drop column defaults, convert columns to text, drop old enum types, update text values, create new enums, cast text back to enum, restore defaults. More steps but works in one migration.
- **Enum renames have massive blast radius.** Renaming PUBLISHED->APPROVED touched 31+ files across schema, services, Zod schemas, API route directories, LLM tools, system prompt, seed data, tests, UI components, and docs. One grep pass isn't enough - do multiple sweeps after changes since some files (like tool-labels.ts) use the enum in non-obvious ways (as object keys, not string literals).
- **Clearing `.next` kills the dev server.** When Next.js caches stale route paths in `.next/types/validator.ts` (e.g. after renaming API route directories), `rm -rf .next` fixes TypeScript compilation errors but kills the running dev server. Always restart with `npm run dev` after.
- **Claude struggles with UI design (but can fix it with research).** Issue #5 UI was "from the 80s" - five layers of near-identical white, no color. Issue #7 fixed it by researching Claude.ai, ChatGPT, Notion, and Lovable for patterns, then writing a UI spec with explicit hex values, WCAG contrast ratios, and component-level styling rules. The spec-driven approach produced a much better result than "make it look modern." Key lesson: give AI concrete design tokens and examples, not vibes.
- **One branch per issue, always.** Issues #11, #13, and #16 all landed on one branch because multiple editor windows shared the same working directory. Got lucky (no conflicts, tests passed), but lost the ability to revert any single issue independently. Fix: use `git worktree` to create separate directories per branch, or work sequentially. Before parallelizing, ask: do they touch the same files? Does one depend on another? Which has the biggest blast radius? Merge the widest-reaching change (like enum renames across 52 files) last.

## Patterns That Work
<!-- Add approaches and conventions that proved effective -->

- **Tag vocabulary for prompts:** `<rules>` (must-follow), `<procedure>` (step-by-step), `<template>` (copy-paste schemas), `<output_format>` (expected output), `<conditions>` (if-then logic), `<guidelines>` (best practices), `<reference>` (lookup tables), `<phase>` (named stages), `<examples>` (good/bad demos).
- **Audit before converting.** Not every file needs XML. Short files, human-facing docs, and already-structured data are fine without it. Focus on files where the AI needs to distinguish content types.
- **Domain commands over raw CRUD.** Instead of generic `PATCH /resource/:id`, use intent-revealing endpoints like `POST /resource/:id/approve`. Each endpoint maps to one business action with clear lifecycle rules. This makes the API self-documenting and prevents invalid state transitions at the routing level.
- **Two-entity versioning.** Split "thing that evolves" into a logical entity (TestProcedure) and immutable version snapshots (TestProcedureVersion). The logical entity holds identity and current status; versions hold content and are immutable once approved. Enforced at service layer with a single-draft rule.
- **Exclusive Arc for polymorphic ownership.** Instead of a generic `parent_type` + `parent_id` (which can't have FK constraints), use one table with optional FK columns per parent type. Zod validates exactly one is non-null. You get real foreign keys and referential integrity.
- **Service layer = transaction boundary.** Every service function wraps its mutation + audit log in a single Prisma interactive transaction. Route handlers stay thin (parse input, call service, return response). This guarantees consistency and makes testing easier.
- **Centralized error handling.** One `handleApiError()` function maps error types to HTTP status codes (ZodError->400, LifecycleError->409, Prisma P2025->404). Route handlers just catch and delegate.
- **Three-model review consensus.** Running `/review-code`, `/ask-gpt`, and `/ask-gemini` in parallel catches different things. Claude caught structural issues, GPT found edge cases, Gemini spotted missing error classes. Fix items all three agree on; track the rest as issues.
- **Triage review findings: fix now vs track later.** After a multi-model review, split findings into quick fixes (do now) and follow-ups (add as comments on existing issues or create new ones). Don't try to fix everything in one pass - it bloats the commit and risks scope creep.
- **Single source of truth for seed data IDs.** Seed scripts should import IDs from the same module the app uses (e.g. `demo-users.ts`), not hardcode duplicates. Caught during review when seed had copy-pasted user IDs that could drift from the source.
- **`Record<EnumA, EnumB>` for type-safe enum mappings.** TypeScript's `Record` with Prisma enum types (e.g. `Record<TestCaseResult, TestCaseStatus>`) ensures every enum value is handled at compile time. Used in the test case result-to-status mapping - if a new enum value is added, TypeScript will error until you handle it.
- **Compact Prisma `select` in LLM tools.** LLM tools must use explicit `select` clauses with `take` limits on nested relations. Without limits, a single traceability chain query could return 40K+ objects (20 x 20 x 5 x 20). With limits (10 x 10 x 3 x 5), max is 1.5K. Also use parallel `prisma.count()` for true totals when paginating.
- **Stable error prefixes for LLM tools.** Return errors as strings with prefixes (`LifecycleError: ...`, `NotFoundError: ...`) instead of throwing. The LLM can parse these to give users plain-language explanations. Use `instanceof` checks (not `error.name` string comparison) for reliable error type detection.
- **Transparent wrappers create a "contrast cascade."** Removing background cards from assistant messages (for modern look) means child elements (code blocks, tables, blockquotes) inherit the warm page background. They need explicit `bg-surface-elevated` + borders. Extract markdown styles to a CSS class (`.chat-markdown`) rather than a 400-char inline Tailwind string.
- **Extract shared utilities early.** Duplicated `str.replace(/([A-Z])/g, " $1").trim()` appeared in 3 files before review caught it. A shared `humanize()` function took 2 minutes to create. Small duplication compounds fast across a growing codebase.
- **Multi-model reviews find orthogonal issues.** Claude review found wrong seed counts and hardcoded hex values. GPT found StatusBadge needed a union type. Gemini found the markdown contrast cascade. Almost zero overlap between the three. Run all three in parallel - it takes the same time as running one.
- **`* { padding: 0; margin: 0 }` kills Tailwind utilities.** A global CSS reset that sets padding/margin to 0 on every element will override Tailwind's `px-4`, `py-2`, etc. because CSS specificity rules mean the wildcard reset applies last. Tailwind v4's `@import "tailwindcss"` already includes Preflight (a sensible reset), so custom resets are redundant. When spacing looks broken everywhere, check globals.css before hunting per-component bugs.
- **Shared state needs a shared store, not props.** When the panel changed from `lg:static` (flex sibling, width handled by flexbox) to always-fixed (overlay), the chat column needed to know the panel width for `padding-right`. Hardcoding the default (540px) broke when users resized. Lifting `panelWidth` into Zustand took 5 minutes and kept both components in sync.
- **Parallel sub-agents are great for audits.** 14 agents covering different domains (CSS, perf, security, services, etc.) found 30+ issues with almost zero overlap. One agent alone would miss entire categories. The key: give each agent a narrow, well-defined scope with specific files to check.
- **Triage audit findings: quick-fix vs. branch-worthy.** The audit found 30+ items. 11 were safe one-liners (selectors, memo, trim, aria attrs). 6 needed design decisions or multi-file changes. Bundling quick fixes into one commit kept momentum; creating issues for the rest prevented scope creep. The split point: "does this change behavior or just improve correctness/polish?"
- **Bare Zustand `usePanelStore()` is a silent perf killer.** Destructuring the whole store (`const { isOpen, content, ... } = usePanelStore()`) subscribes to every property change. During drag-to-resize, that means re-rendering the entire panel 60 times per second. Individual selectors (`usePanelStore(s => s.isOpen)`) subscribe only to what you use. Same API, wildly different performance.
- **Audit findings need context, not just correctness.** Issue #34 had three "medium" audit findings (network retry, abort on user switch, message scan optimization) that were all technically valid patterns. But none were actual problems: retries solve network failures that don't happen in a demo app, the abort race condition only exists because of the demo user picker (real auth means page reloads), and the "O(n^2)" scan was really O(n) with early exits on 50-100 messages. Lesson: always ask "is this a real problem for *this* app?" before building the fix. Audits check against general best practices - they don't weigh practical relevance.
- **Reverse cascade operations mirror forward cascades structurally.** Issue #54 (reactivation) is the exact reverse of cancellation. Same cascade chain (PR -> SR -> TP -> TC), same idempotency guards (skip already-active instead of skip already-canceled), same audit pattern. When building an "undo" operation, start by reading the forward operation and mirror its structure. The cascade helpers, transaction boundaries, and guard patterns all transfer directly.

- **Manual QA scenarios that share DB state affect each other.** Reactivating a CANCELED entity in Scenario 2 meant Scenario 8's negative test target was no longer CANCELED. When designing multi-scenario walkthroughs, either reset between scenarios or track cumulative state changes and adjust later scenarios accordingly.
- **"Cancel" is ambiguous in AI chat UIs.** A user saying "no, cancel" (meaning "nevermind") can be misinterpreted as a domain cancel command. Destructive operations need explicit entity references - bare "cancel" should default to aborting the current action, not executing a lifecycle transition.
- **Mirror operations should clear the same fields.** Re-execute properly clears all execution data (result, notes, executor, executedAt) when resetting a TC to PENDING. Reactivation does the same status reset but leaves stale results. When two paths lead to the same target state, they should produce identical field states.

- **Server-driven metadata beats client-side business logic.** Instead of hardcoding "DRAFT PR can be approved or canceled" in React components, compute `availableActions` and `editableFields` on the server and return them as metadata. The panel renders whatever the server says - no lifecycle rules in the UI. When rules change, update one function (`computeAvailableActions`), not every component.
- **Dead code can hide behind integration gaps.** Issue #10 wired up clickable table rows end-to-end (click handler, loading state, navigation, entity type resolution) across 3 files, but the feature never worked because the table queries didn't include an ID column in the row data. 5 reviewers read the code and it looked correct in each file. Only the code quality review caught the cross-layer gap by tracing the full data flow from Prisma query to React render. When adding interactive features, test the full pipeline, not just each layer.
- **Running 7 reviews in parallel is worth the token cost.** 5 internal reviews (UX, code, plan compliance, pre-release, browser QA) + 2 multi-model debates (GPT, Gemini) found ~35 issues with very little overlap. The UX review caught focus ring gaps. The code review caught the dead navigation feature. The pre-release review caught the TPV phantom title field. The GPT debate caught race conditions. One reviewer would have missed most of these. Total time: ~10 minutes wall clock (parallel), vs ~60+ minutes serial.
- **Edit mode toggle is simpler than click-to-edit for a V2.** Click-to-edit (Notion-style) needs per-field state, blur-save race conditions, and per-field error handling. Edit mode toggle needs one boolean. For a learning codebase where you're already managing navigation history + lifecycle actions + chat notifications, the simpler pattern saved significant complexity.

- **Deduplicate sources of truth when you touch both files.** The user-picker had a duplicated user list "to avoid importing server-side code." But the source file (`demo-users.ts`) had no server-side imports - the duplication was unnecessary from day one. When a task touches both the source and the copy, that's the ideal time to eliminate the copy. If you don't, the next person who adds a user will update one file and forget the other.
- **Comments and summary counts drift from actual data.** When rewriting seed data, the inline comments ("Status mix: 7 APPROVED, 2 DRAFT, 1 CANCELED") and console.log summary were copy-pasted from the old version but never recounted against the new data. The actual data had 8/1/1. Always count from the data arrays, not from the comments. A quick `grep -c` after writing catches this in seconds.
- **Seed timestamp consistency matters for audit trails.** TC9's `executedAt` was set to Day 18, but the audit log SKIP entry was Day 19 (cancellation day). The app renders both fields - if they disagree, the UI tells two different stories about when something happened. When writing seed data with paired entity fields and audit entries, cross-check the timestamps match.
- **Research the domain before writing fake data.** 10 minutes of Pokedex research (Bulbapedia, game wikis) produced requirements grounded in real features - scanner, cry playback, habitat maps, clamshell form factor. Without research, the data would have included features the Pokedex never had (messaging, gym finder, GPS). Fact-checked data is more fun and more believable than invented data.
- **7 parallel reviews find issues that 1 review misses, but deduplication is key.** Running UX, code, plan, pre-release, browser, Gemini, and GPT reviews in parallel produced ~25 unique findings from ~40 total (significant overlap on stale docs and count mismatches). The value is in the non-overlapping finds: code review caught the TC9 timestamp, pre-release caught the exact count discrepancies, browser review caught the middleware comment. Deduplicate before acting - otherwise you fix the same thing 4 times.

- **Background dev servers in CI/automation need health checks.** Starting `npm run dev` in the background doesn't mean it's ready - Next.js needs time to compile. A `curl` to `localhost:3000` will hang until compilation finishes. Always poll with a timeout or wait for the "Ready" log before running tests against the server. Also, killing a dev server by port (`lsof -ti:3000 | xargs kill`) is more reliable than killing by process name, since Next.js spawns multiple child processes.
- **In-memory rate limiting doesn't work on Vercel serverless.** Each function invocation gets its own memory space - the `Map` storing request timestamps doesn't persist between invocations. It still provides per-instance throttling (some protection), but for production-grade enforcement you need a persistent backend (Vercel KV, Upstash Redis, or a DB table) for global rate limiting on serverless.
- **`string.length` measures UTF-16 code units, not bytes.** A body size check using `rawBody.length` undercounts multi-byte characters. Use `new TextEncoder().encode(rawBody).byteLength` for actual UTF-8 byte length. Small difference, but an attacker could exploit it to bypass size limits with multi-byte payloads.
- **Generic error messages prevent schema leakage.** Prisma error messages include table names, constraint names, and field paths (e.g. "Unique constraint failed on the fields: (testProcedureId, versionDraft)"). Returning `error.message` as-is teaches attackers your schema. Map to generic messages ("Record not found", "Bad request") and log the details server-side.
- **`playwright-core` is not bundled - install it and the browser separately.** `playwright-core` is the bare library without bundled browsers. After `npm install playwright-core`, you also need `npx playwright-core install chromium`. Without the browser binary, `browse.js` fails with a confusing "Executable doesn't exist" error. The two-step install is by design (keeps the package small), but it's easy to forget.
- **Token-driven theming is the highest-leverage UI change.** Swapping ~15 CSS custom properties in one `@theme` block rippled through every component automatically. The bespoke per-component changes (Pokeball buttons, battle phrases) took 10x the effort for maybe 2x the visual impact. Start with tokens - you get 80% of the visual shift from 20% of the work.
- **Frosted glass and retro don't mix.** Glassmorphism (blur, transparency) signals "modern/sleek" while thick borders signal "retro/solid." Mixing them creates visual confusion. When theming, commit to one visual language fully rather than layering two. This wasn't obvious until we saw them side by side in the HTML prototype.
- **HTML prototypes beat discussion for visual decisions.** A double-clickable HTML file with 3 palette variants resolved more design questions in 30 seconds than text conversation did in 10 minutes. For any visual decision where the answer is "I need to see it," build a standalone prototype first.
- **Semantic color token collisions are silent bugs.** When your brand color IS red, `--color-primary` and `--color-danger` silently become the same hex. Passes every automated test but fails the first time a user sees an error next to a button. After setting a palette, always diff the semantic tokens against each other.

- **Claude Code's Write tool requires a Read-first guard, even for new files.** The Write tool rejects creating a file unless you've Read it first in the same session. For new files, Read returns "file does not exist" - but that error satisfies the guard. Background subagents enforce this strictly while the main thread is more lenient. When writing slash command prompts that create files, always instruct "Read first (ignore the error), then Write." Also avoid Bash `>` redirects in prompts meant for subagents - use the Write tool instead.
